# 开发文档

## 1. 项目判断

这个仓库不是纯前台，也不是纯后台，而是三层混合体：

1. 旧站基础能力
2. 新迁移的多模板前台
3. 用于配置模板内容的后台主题系统

当前最接近“完成”的部分，是主题内容已经可以在后台维护并回流到前台。  
当前最没收口的部分，是“整站模板切换”还没有被抽成一个统一开关。
当前已经基本打通的是首页、商品列表、商品详情、购物车、结账、搜索、页头、页脚和分类入口的真实数据链路。

- `app/(shop)/(other-pages)/`：常规静态页面

### 旧站副本

- `archive/siteCopy/`：旧站页面副本归档

这部分是迁移时保留下来的对照实现，当前定位为旧站归档副本：适合做回退参考，不再作为新功能主链路继续扩张，也不再参与生产路由构建。详见 `archive/siteCopy/README.md`。

### 新模板素材

- `app/newSite-components/`：模板组件库
- `app/newSite-data/`：模板用静态数据

- `templates/`：前台模板逻辑层，当前启用模板是 `templates/modave/`

## 3. 数据流

### 首页

1. `app/(shop)/page.jsx`
2. 调用 `shared/api/site.js`
3. 请求 `/api/site/home` 和 `/api/site/categories`
4. `app/api/site/home/route.js` 从 `StoreTheme` 和 `Product` 组装首页数据
5. `templates/modave/HomePage.jsx` 渲染当前模板首页

### 页头

1. `app/components/theme/header/header.js`
2. 请求 `/api/site/header`
3. `app/api/site/header/route.js` 读取 `StoreTheme`
4. `app/components/theme/header/headerData` 负责渲染

### 商品详情

1. `app/(shop)/(productDetails)/product-detail/[id]/page.jsx`
2. 调用 `shared/api/products.js`
3. 请求 `/api/site/product/get-single`
4. `app/api/site/product/get-single/route.js` 查商品、分类、关联商品
5. `templates/modave/ProductDetailPage.jsx` 渲染当前模板商品详情

### 商品列表

1. `app/(shop)/(products)/shop-default-grid/page.jsx`
2. 静态引入当前启用的 Modave 模板页面
3. `templates/modave/ShopPage.jsx` 渲染当前模板商品列表外壳
4. `app/newSite-components/products/Products1.jsx` 在客户端请求 `/api/site/product` 并处理筛选、排序、分页

### 购物车

1. `app/(shop)/(products)/shopping-cart/page.jsx`
2. 静态引入当前启用的 Modave 模板页面
3. `templates/modave/CartPage.jsx` 渲染当前模板购物车外壳
4. `app/newSite-components/otherPages/ShopCart.jsx` 使用 `newSite-context` 管理购物车展示和操作

### 结账

1. `app/(shop)/(products)/checkout/page.jsx`
2. 静态引入当前启用的 Modave 模板页面
3. `templates/modave/CheckoutPage.jsx` 渲染当前模板结账外壳
4. `app/newSite-components/otherPages/Checkout.jsx` 继续处理结账表单、库存校验、订单创建和支付

### 搜索

1. `app/(shop)/(products)/search-result/page.jsx`
2. 静态引入当前启用的 Modave 模板页面
3. `templates/modave/SearchPage.jsx` 渲染当前模板搜索外壳
4. `app/newSite-components/products/SearchProducts.jsx` 继续处理搜索参数、搜索请求和结果展示

### 常规页面

1. `app/(shop)/(other-pages)/*/page.jsx`
2. 静态引入当前启用的 Modave 模板页面
3. `templates/modave/*Page.jsx` 渲染当前模板页面
4. 模板页面继续复用 `app/newSite-components/otherPages/` 的页面组件

### 账号中心

1. `app/(shop)/(my-account)/*/page.jsx`
2. 静态引入当前启用的 Modave 模板页面
3. `templates/modave/MyAccount*.jsx` 渲染当前模板账号页面
4. `templates/modave/MyAccountShell.jsx` 统一账号中心公共布局
5. `app/newSite-components/my-account/` 继续处理账号页面具体内容

### 后台主题保存

1. `app/admin/theme/*`
2. 通过 `useSubmitThemeForm`
3. 请求 `/api/admin/store/theme`
4. 写入 `StoreTheme`

## 6. 目前最像“模板切换”的地方

现在已经有一个最小模板注册表：`templates/registry.js`。

它默认读取 `NEXT_PUBLIC_ACTIVE_TEMPLATE`，没有配置时使用 `modave`。这还不是完整的后台模板切换器，目前主要承担模板元信息登记；生产路由仍静态引入当前模板页面，这是“部署时只选一套模板”场景下更稳、更轻的做法。

项目里原本更接近模板切换的，是：

- 不同首页样式页面曾分散在 `app/(shop)/(homes)/`，现在已归档到 `archive/modave-demos/templates/modave/examples/homes/`
- 商品详情页有很多展示模板分支
- 主题内容可以在后台切换不同区块数据

所以现在更准确的说法是：

> 这是一个“多模板素材已迁入、主题配置已联通、整站模板切换还未统一收口”的系统。

## 7. 推荐的后续收口方式

### 方案 A：最小改动

增加一个站点级字段，例如：

- `activeTemplate`
- `homeTemplate`
- `productTemplate`
- `headerTemplate`

优点是改动小，适合快速上线。

### 方案 B：模板注册表

建立一个模板配置中心：

- 模板名
- 对应首页组件
- 对应商品详情组件
- 对应 header/footer 组件
- 对应默认数据源

优点是后期扩展最稳，适合长期维护。

## 8. 开发约定

- 新页面优先放在 `app/(shop)/` 的正式路由里
- 正式路由只负责取数、鉴权、SEO 和选择模板
- 模板页面优先放到 `templates/<template-id>/`
- 多模板共享请求放到 `shared/api/`
- 模板静态数据先放 `app/newSite-data/`，后续逐步迁到具体模板目录
- 现有可复用 UI 仍放 `app/newSite-components/`，稳定后再迁到具体模板
- 后台配置统一写入 `StoreTheme`
- 前台优先通过 API 取数，不直接硬编码
- 默认使用本地字体构建，避免生产构建依赖 Google Fonts；需要 Google Fonts 时设置 `NEXT_PUBLIC_FONT_PROVIDER=google`
- 支付默认走当前站点；跨站支付必须显式设置 `NEXT_PUBLIC_ENABLE_CROSS_SITE_PAYMENT=true` 并配置 `NEXT_PUBLIC_PAYMENT_SITE_URL`
- 首页风格优先做成当前模板内的首页变体，通过 `NEXT_PUBLIC_MODAVE_HOME_VARIANT` 切换；只有整站页面都不同才新增完整模板

## 10. Demo 路由清理原则

- 当前生产路由只保留实际启用模板需要的页面。
- 模板示例页移动到 `archive/modave-demos/templates/<template-id>/examples/`。
- 不直接删除迁移进来的示例，先归档，等确认无用后再清理。
- 恢复某个示例为生产页面时，先改造成模板组件，再通过 `app/(shop)` 正式路由接入。
- 生产布局不展示 DemoModal，避免暴露已归档 demo 链接。
